# Code Review Rule (Tutor)

## 목표
- 리뷰 출력 형식과 판단 기준을 고정해서, 매 리뷰 품질(일관성)을 유지한다.
- "강점/약점/학습 방향성"은 이번 제출물(현재 diff)에서 관찰 가능한 사실만을 근거로 작성한다.
- 커맨드 입력에 따라 "단일 경로 리뷰" 또는 "비교(품질 변화) 포함 리뷰"를 수행한다.

## 출력 형식(강제)
- 반드시 아래 섹션을 이 순서로 출력한다:
  0) 범위
  1) 결론(상태+근거)
  2) Blocking(필수 수정)
  3) Non-blocking(권장)
  4) 학습 포인트 태그
  5) 강점(이번 제출물 기준)
  6) 약점(이번 제출물 기준)
  7) 학습 방향성(우선순위 1~2개)
  8) 코드 퀄리티 변화 분석(@path2가 제공된 경우에만)

## 증거 규칙(강제)
- 모든 지적/평가는 (파일경로:라인범위) 또는 (함수/클래스 이름)로 근거를 명시한다.
- 근거를 특정할 수 없으면 "근거 불충분"으로 적고 Blocking으로 올리지 않는다.
- 강점/약점/학습 방향성도 동일하게 근거를 요구한다.
- diff에 없는 사실/의도/능력 추정 금지 ("아마", "성향상", "원래" 등 금지)

## 심각도 규칙(강제)
- Blocking으로 분류 가능한 조건:
  - 컴파일 오류 가능성
  - 런타임 크래시/UB 가능성
  - 요구사항/스펙 위반
  - 보안/정보 노출/권한 문제
  - 과제/수업에서 금지한 패턴 위반
- 그 외는 Non-blocking

## 금지
- 과도한 스타일 지적(의미 없는 컨벤션 지적) 금지
- 바꾸지 않아도 되는 리팩터링 제안 금지(명확한 이유 없으면)
- 개인 평가/감정 표현/동기부여 문장 금지 ("잘하고 있다", "센스가 있다" 등 금지)
  - 대신 "관찰→근거→영향"으로만 기술

## 톤
- 사실/근거/영향/수정방향 중심. 단정은 근거가 있을 때만.

## 섹션별 작성 규칙(강제)

### 5) 강점(이번 제출물 기준)
- 최대 3개만 작성.
- 각 항목은 아래 형식을 따른다:
  - 관찰: (무엇을 잘했는지, 코드 수준 사실)
  - 근거: (파일:라인 또는 함수/클래스)
  - 왜 강점인지: (품질 기준: 안정성/가독성/확장성/성능/테스트 용이성 중 무엇에 기여하는지)

### 6) 약점(이번 제출물 기준)
- 최대 3개만 작성.
- 각 항목은 아래 형식을 따른다:
  - 관찰: (문제/부족한 점, 코드 수준 사실)
  - 근거: (파일:라인 또는 함수/클래스)
  - 영향: (버그/유지보수/성능/스펙 리스크 등)
  - 성격: (Blocking인지 Non-blocking인지)

### 7) 학습 방향성(우선순위 1~2개)
- "학습 과제 나열"이 아니라, 약점/이슈를 바탕으로 한 **우선순위 개선축**만 제시한다.
- 각 항목은 아래 형식을 따른다:
  - 목표: (한 문장, 측정/판정 가능하게)
  - 이번 코드에서의 근거: (파일:라인 또는 함수/클래스)
  - 다음 제출물에서 체크할 기준: (PASS/FAIL로 판정 가능하게 1~2개)
